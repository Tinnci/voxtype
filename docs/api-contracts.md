# VoxType API contracts

This document records the implemented internal and public API boundaries.
Names may evolve during version 0.x, but responsibilities and dependency
direction should remain stable.

## 1. Domain model

The domain layer depends only on the Rust standard library.

```rust
pub struct SessionId(String);

pub enum SessionState {
    Idle,
    Preparing,
    Listening,
    Finalizing,
    Inserting,
    Completed,
    Cancelled,
    Failed,
}

pub enum TriggerMode {
    PushToTalk,
    Toggle,
}

pub struct StartRequest {
    pub mode: TriggerMode,
    pub profile: Option<String>,
    pub target_policy: TargetPolicy,
}

pub enum Command {
    Start(StartRequest),
    Stop { session: SessionId },
    Cancel { session: SessionId },
    Shutdown,
}

pub enum Event {
    StateChanged { session: SessionId, state: SessionState },
    PartialTranscript { session: SessionId, text: String },
    FinalTranscript { session: SessionId, text: String },
    Completed { session: SessionId, insertion: InsertionResult },
    Failed { session: SessionId, error: AppError },
}
```

These examples describe shape, not a commitment to expose public Rust library
types in version 0.1.

## 2. Application controller

`voxtype_app::AppController` is the single owner of the state machine, provider
registry/runtime state, lifecycle event queue, and terminal-result cache. It is
used from a standard thread without an async runtime.

```rust
impl AppController {
    fn apply(&mut self, command: Command) -> Result<CommandEffect, VoxError>;
    fn drain_events(&mut self) -> Vec<AppEvent>;
    fn plan_route(&self, policy: &RoutingPolicy, now: Instant)
        -> Result<RoutePlan, VoxError>;
    fn record_route_result(
        &mut self,
        result: &RecognitionRouteResult,
        audio_millis: u64,
        now: Instant,
    );
}
```

Implementation guidance:

- Use bounded channels for audio, provider results, lifecycle events, and wake
  notifications.
- A successful `apply` means accepted by the state machine, not merely queued.
- Session IDs are generated by the application, never by an ASR provider.
- Cancellation is idempotent for the current session.
- Only one recording session exists in the MVP.

## 3. Audio boundary

```rust
pub struct AudioFormat {
    pub sample_rate_hz: u32,
    pub channels: u16,
    pub sample: SampleFormat,
}

pub enum SampleFormat {
    I16Le,
    F32,
}

pub struct AudioChunk {
    pub sequence: u64,
    pub captured_at_micros: u64,
    pub format: AudioFormat,
    pub samples: Vec<u8>,
}

pub trait CaptureAdapter: Debug + Send + Sync {
    fn start(&self, device: Option<&str>)
        -> Result<Box<dyn CaptureSession>, VoxError>;
}

pub trait CaptureSession: Debug + Send + Sync {
    fn stop(self: Box<Self>) -> Result<CapturedAudio, VoxError>;
    fn cancel(self: Box<Self>);
    fn drain_metrics(&mut self) -> Vec<CaptureFrameMetrics>;
}
```

Rules:

- Desktop/provider modules never receive cpal or PipeWire native types.
- The current production port returns a completed private PCM spool plus
  transcript-free frame metrics. `AudioChunk` remains the neutral type for the
  future bounded live-stream extension.
- Capture timestamps are monotonic offsets, not wall-clock timestamps.
- The audio adapter owns device-format conversion; the provider adapter owns
  provider-specific framing and encoding.
- Queue saturation is explicit (`drop`, `block`, or `fail`) and measurable.

## 4. ASR provider boundary

`voxtype_app::ProviderAdapter` is the only runtime recognition contract. It is
independent of Tokio, WebSocket, zbus, and provider-specific types.

```rust
pub trait ProviderAdapter: Send + Sync {
    fn id(&self) -> &ProviderId;
    fn capabilities(&self) -> ProviderCapabilities;
    fn recognize(
        &self,
        input: RecognitionInput<'_>,
        cancellation: &CancellationToken,
    ) -> Result<ProviderSuccess, ProviderAttemptFailure>;
}
```

All current adapters consume a completed PCM recording. A future live-stream
extension will add a bounded audio source and recognition event sink to this
same application boundary; it will not reintroduce a second provider trait.

Rules:

- Persisted configuration stores only secret references. Concrete adapters
  resolve opaque secret values per attempt; secret material is absent from
  recognition requests, errors, and debug output.
- Provider error categories are stable: configuration, authentication,
  permission, connection, timeout, protocol, rate limit, unavailable, cancelled,
  and internal.
- Provider raw JSON/protobuf is available only behind a redacted diagnostic mode.
- Runtime orchestration is tested with fake adapters and private temporary PCM;
  protocol adapters use loopback transports and synthetic audio.

### Provider registry and routing

```rust
pub struct ProviderId(String);

pub struct ProviderRegistry {
    adapters: BTreeMap<ProviderId, Arc<dyn ProviderAdapter>>,
}

impl ProviderRegistry {
    fn register(&mut self, adapter: Arc<dyn ProviderAdapter>)
        -> Result<(), VoxError>;
    fn get(&self, id: &ProviderId) -> Option<&Arc<dyn ProviderAdapter>>;
}

pub trait ProviderRouter {
    fn plan(
        &self,
        policy: &RoutingPolicy,
        health: &BTreeMap<ProviderId, ProviderHealth>,
    ) -> Result<RoutePlan, VoxError>;
}

pub struct RoutePlan {
    pub providers: Vec<ProviderId>,
    pub replay: ReplayPolicy,
}

pub enum ReplayPolicy {
    Never,
    BeforeAudioAccepted,
    BufferedWithConsent,
}

pub enum AudioAcceptance {
    NotAccepted,
    PossiblyAccepted,
    Accepted,
}

pub struct ProviderAttemptFailure {
    pub error: VoxError,
    pub transport_started: bool,
    pub audio_acceptance: AudioAcceptance,
}
```

Routing rules:

- Provider selection currently uses configured order, registered availability,
  recent cooldown health, and replay policy. Capabilities are exposed by the
  adapter contract for validation and the future live-stream extension;
  cost/latency ranking is not implemented.
- Provider-specific errors are normalized before the router sees them.
- Authentication and invalid-configuration failures do not fall back silently;
  the user must know that the configured provider is unusable.
- Connection/unavailable/rate-limit failures may use an allowed fallback.
- Once audio has been accepted, replay to another cloud requires the profile's
  explicit `BufferedWithConsent` policy.
- Cancellation and connection loss after transport start may leave acceptance
  ambiguous. `PossiblyAccepted` is treated like `Accepted`; uncertainty never
  authorizes replay. Curl upload-byte evidence may prove `NotAccepted`, while a
  successful provider response proves `Accepted`.
- Parallel racing and result voting are future opt-in modes, never defaults.
- The provider registry contains no dynamic library loading in version 0.x.

### Provider verification requirements

The shared application suite covers registry identity, cancellation before
dispatch, ordered fallback, replay consent, empty final transcripts, and
lifecycle evidence. Each concrete adapter additionally covers the applicable
items below with loopback or process fixtures:

- advertised capabilities match accepted requests;
- cancellation terminates send and receive paths;
- bounded backpressure is respected;
- partial/final ordering is normalized when a future live-stream adapter
  advertises partial results;
- timeout and error categories are stable;
- pre-transport, possible-upload, and accepted-audio failures retain accurate
  lifecycle evidence for routing and usage accounting;
- secrets and transcript content are redacted from normal diagnostics;
- an empty or malformed provider response cannot become inserted text;
- provider shutdown leaves the daemon ready for the next session.

## 5. Text insertion boundary

```rust
pub trait InsertionAdapter: Debug + Send + Sync {
    fn arm(
        &self,
        mode: InsertionMode,
        session: &SessionId,
    ) -> Result<InsertionArm, VoxError>;
    fn commit(
        &self,
        arm: &InsertionArm,
        text: &str,
    ) -> Result<InsertionOutcome, VoxError>;
    fn cancel(&self, arm: &InsertionArm);
}
```

The adapter arms the intended target before capture. `InsertionArm` contains
only the session ID and selected backend; Fcitx retains target identity and
secure-field state inside its private bridge. Automatic fallback may select
copy-only only when Fcitx transport/runtime is unavailable. Focus and
secure-field rejection remain terminal.

## 6. Secret storage boundary

```rust
pub struct SecretRef(String);

pub trait SecretStore: Send + Sync {
    fn get(&self, reference: &SecretRef) -> Result<SecretBytes, SecretError>;
    fn put(&self, label: &str, secret: SecretBytes) -> Result<SecretRef, SecretError>;

```

`SecretBytes` must redact `Debug`/`Display` and zero memory on drop if a small,
well-maintained dependency is accepted for that purpose.

### Signals

```text
StateChanged(s state, s session_id)
SessionFinished(s session_id, s outcome, s error_code, s backend, t char_count)
```

`ProviderStatus` returns a versioned JSON snapshot. A configured credential or
an open circuit is not proof that a provider works. `verified=true` is emitted
only after a real successful provider response and expires after the advertised
TTL. Failures retain stable category/code metadata without exposing response
bodies or credentials.

```json
{
  "schema": 1,
  "providers": {
    "cloud": {
      "route_available": true,
      "verified": true,
      "verified_age_seconds": 12,
      "verification_ttl_seconds": 900,
      "consecutive_failures": 0,
      "retry_after_seconds": null,
      "last_failure_age_seconds": null,
      "last_error_category": null,
      "last_error_code": null
    }
  }
}
```

`CheckLastGrammar` and `CheckContextGrammar` return a versioned cleanup-report
JSON document. Edits use byte offsets that are guaranteed to lie on UTF-8
character boundaries. Each edit includes the exact original span, replacement,
stable rule ID, confidence, and either `safe` or `review` safety. The report
also includes an FNV-1a fingerprint and source length; a future apply operation
must reject the report when the complete source no longer matches.

```json
{
  "schema": 1,
  "clean": false,
  "original_fingerprint": "fnv1a64:...",
  "original_bytes": 18,
  "original_characters": 18,
  "suggested": "Hello, world!",
  "safe_edit_count": 2,
  "review_edit_count": 2,
  "edits": [
    {
      "start_byte": 0,
      "end_byte": 2,
      "original": "  ",
      "replacement": "",
      "rule_id": "cleanup.outer-whitespace",
      "message": "Remove leading horizontal whitespace",
      "safety": "safe",
      "confidence": "high"
    }
  ]
}
```

The Fcitx context form adds a `source` object containing only frontend,
application identity, generation and truncation metadata. Password/sensitive
contexts are still rejected inside the addon before text reaches Rust.

`SessionResult` reads a bounded cache of the latest 32 terminal sessions. The
cache contains the same metadata as `SessionFinished` and never contains
transcript text. Repeating a query is side-effect free. A missing entry means
the session is still active, unknown, or older than the cache window.

`StateChanged` contains lifecycle metadata only and is emitted in the exact
order accepted by the daemon state machine. It never contains transcript text
or provider secrets. `SessionFinished` is emitted once after the terminal state
and contains only outcome metadata: `completed`, `no-speech`, `cancelled`, or
`failed`; `error_code`, insertion backend, and character count are empty/zero
when not applicable.

An empty profile selects `default_profile`; an empty session ID selects the
currently active session. `UsageStatus` is JSON because the counters and soft
quota fields are additive during version 0.x. Transcript history is memory-only,
limited to twenty VoxType commits, and is never a general application-input
history.

`Stop` acknowledges the transition to `finalizing` after capture/VAD work has
been accepted. Provider I/O runs in a cancellable background job; completion or
failure is reflected by the ordered signals, while `Cancel` and status queries
remain responsive. CLI callers may use `voxtype stop --wait [SESSION]`; it polls
the retained terminal result with a five-minute bound, so a missed or failed
signal cannot hang the caller. `voxtype result SESSION` performs a single
query. A worker result is applied only when its opaque session ID still matches
the active finalizing session.

Transcript and partial-result signals are intentionally absent until observer
privacy and same-user access semantics are defined.

`CheckContextGrammar` is an explicit user action. It reads at most 4096
characters from the focused Fcitx context, prefers the current selection, and
otherwise reviews at most 1200 characters before the cursor. Sensitive fields
are rejected by the addon and the snapshot is never stored in transcript
history.

## 8. CLI mapping

The CLI is a thin D-Bus client. It contains no audio, provider, credential, or
insertion implementation.

```text
voxtype start [PROFILE]
voxtype stop [SESSION]
voxtype stop --wait [SESSION]
voxtype toggle [PROFILE]
voxtype cancel [SESSION]
voxtype status
voxtype providers
voxtype usage
voxtype result SESSION
voxtype grammar context|last|show|history|clear
voxtype fcitx-context
voxtype doctor [audio|shortcut|provider|insertion|all]
voxtype config path|validate
voxtype secret set NAME
```

## 9. Configuration schema

The top-level TOML sections are:

```toml
schema_version = 1
default_profile = "default"

[daemon]
max_recording_seconds = 120
log_level = "info"

[desktop]
target_policy = "strict"
insertion_backend = "auto"
restore_clipboard = true

[audio]
device = "default"
queue_chunks = 50

[profiles.default]
provider = "doubao-unofficial"
language = "zh-CN"

[profiles.default.routing]
fallbacks = []
replay = "never"

[providers.doubao-unofficial]
credential = "secret-service://voxtype/doubao/default"
response_timeout_seconds = 15
```

Secrets, transcripts, device registration bodies, and raw provider tokens are
never valid configuration values.
